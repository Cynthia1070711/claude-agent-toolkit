- **建檔者**：GC-PRO (Gemini CLI 協調器)

## 摘要
本報告探討並彙整了多 Agent 針對「Agent 上下文架構原理與 Token 減量策略」的深入討論，重點著眼於如何透過導入檢索增強生成 (RAG) 模式，以技術知識庫及上下文記憶庫取代全專案檔案的粗放讀取。此策略的核心不僅在於降低 Token 成本，更在於提升 Agent 的上線效率與降低重複錯誤的發生率。

---

## 1. 建立上下文資料庫（記憶上下文、快速搜尋）

**架構觀點 (Winston - 架構師)：**
大型語言模型的上下文視窗有限，將原始專案檔案塞入 Prompt 中，在長期擴展性上屬於反模式。
- **解決方案**：導入 RAG 模式。不需複雜的新技術棧，採用如 Chroma 或 PostgreSQL 的 pgvector 等穩定的向量資料庫 (Vector Database) 配合標準 Embeddings 即可達成。
- **作法**：將過去的對話日誌、決策紀錄按語意及時間戳記進行索引。當 Agent 啟動任務時，僅依據任務關鍵字進行相似度檢索，拉取最相關的結果作為 Prompt。如此可在保留深層專案記憶的同時，大幅降低 Token 消耗。

**流程觀點 (Wendy - 工作流建構大師)：**
從工作流角度出發，效率與狀態管理是首要考量。不應讓 Agent 於任務中途隨意查詢資料庫，以免引發延遲或不可預期的錯誤狀態。
- **準備狀態**：需將檢索過程前置化。
  1. **觸發**：使用者發起任務（如 `dev-story`）。
  2. **上下文擷取**：在主要執行 Agent 被呼叫前，先由輕量級的檢索工作流分析使用者關鍵字，查詢向量資料庫並格式化結果。
  3. **執行**：將檢索到的高價值上下文精確注入到 Prompt 中（例如 `<historical_context>` XML 區塊）。
- **容錯機制**：當資料庫服務不可用時，必須具備優雅降級能力，回退至標準的讀檔模式，確保工作流不中斷。

## 2. 建立專案技術資料庫（技術與經驗傳承）

**商業價值觀點 (John - 產品經理)：**
導入技術庫的終極商業價值在於**加快 Agent 上線速度**及**降低缺陷率**。
- 若 Agent 重蹈覆轍，使用了過去已經驗證為失敗的技術實驗，將會浪費寶貴的時間與成本。
- **內容收錄**：專案架構決策紀錄 (ADR)、Bug 修復歷程、以及失敗的技術嘗試等。這些資料不僅是成功的指引，更是避坑的指南。
- 將過往「不可行方式」與「成功案例」結構化儲存後，可作為開發時的自動防呆機制，這也是本策略中最具 MVP (最小可行性產品) 價值的切入點。

---

## 3. 其他高價值延伸應用情境

若記憶庫與技術庫的基礎建設完成，可將其潛力發揮至以下關鍵情境：

1. **新進人員/Agent 培訓模擬器**
   - 剛加入專案的開發者（無論是人類還是新部署的 AI Agent）可直接查詢：「專案目前的身分驗證 (Auth) 是如何處理的？」並立即從技術資料庫獲得綜合彙整的權威解答，大幅減少翻閱程式碼的時間。
2. **自動化程式碼審查攔截**
   - 在執行 `code-review` 工作流時，系統可自動比對 PR 中的程式碼模式。若發現與過去被標記為「失敗作法」或「效能瓶頸」的歷史紀錄相符，即可自動提出警告，提升審查質量。
3. **自動化 Sprint 回顧生成**
   - 上下文資料庫可分析整個 Sprint 期間所有 Agent 的對話及錯誤日誌，自動產生瓶頸分析報告，指出哪些任務消耗了最多的 Token 或遇到最多次重試，藉此持續優化 Agent 的溝通與協作模式。

---

> **結論**：實作 Agent 專用記憶庫與技術資料庫，不僅是技術層面的優化方案（減少 Token 與成本），更是系統化保存專案智慧財產、提升多 Agent 協同作戰能力的戰略性基礎建設。建議優先以「專案技術資料庫」作為第一階段 MVP 進行實作驗證。
