### 0.0 BMAD 版本與配置檢查（Critical — 首次執行）

```yaml

  1. _bmad/ 資料夾存在
     - 確認 _bmad/_config/manifest.yaml 存在
     - 讀取 installation.version 確認版本 (應為 v6.0.0-alpha.21 或更新)

     - 檢查 project_name 是否正確

     .claude/ 與 .gemini/ 資料夾:
       - ✅ 兩者都支援 @_bmad/... 引用語法（通用）
       - ❌ Gemini 不支援 @file 語法（Claude 專屬）
       - ✅ BMAD v6 已統一使用 @_bmad/core/tasks/workflow.xml 作為執行引擎

     - 若使用 Playwright，確認是否執行過 `npx playwright install`
```

## PART C: 銜接模式 (Resume & Handover Mode)

> **適用情境**:
> 1. 開新視窗繼續已有工作
> 2. 跨引擎切換（例如：Claude 規劃 → Gemini 實作）

### C.1 快速上下文載入（< 5 秒）

```yaml
必讀檔案 (依序):
  0. _bmad/bmm/config.yaml（取得 output_folder 路徑）
  1. {output_folder}/project-context.md
  2. CLAUDE.md 或 GEMINI.md（根據當前 CLI）
  3. {output_folder}/tracking/active/*.md（進行中任務）
  4. {output_folder}/technical-decisions/README.md
```

### C.2 跨引擎交接協議（Handoff Protocol）

**三步驟驗證**（任何引擎接手任務前必須執行）:

```yaml
Step 1 - 讀取全域狀態:
  檔案: docs/implementation-artifacts/sprint-status.yaml
  目的: 確認目標 Story 的狀態正確
  驗證: 狀態應為預期值 (如 dev-story 接手時應為 ready-for-dev)

Step 2 - 讀取追蹤檔案:
  路徑: docs/tracking/active/{story-id}.track.md
  目的: 確認追蹤檔案存在且內容完整
  驗證: 上一階段的任務清單已完成

Step 3 - 確認最後一條 log:
  位置: 追蹤檔案底部的執行日誌
  目的: 確保不重複執行已完成步驟
  格式: [AGENT-ID] ISO-8601 動作摘要
```

### C.3 共享狀態機設計

```yaml
共享狀態檔案（所有引擎共用）:
  sprint-status.yaml: 所有 Story 的狀態總覽
  docs/tracking/active/*.track.md: 單一 Story 的詳細執行日誌
  docs/project-context.md: 專案級上下文

非共享狀態（各引擎獨立）:
  .claude/: 僅 Claude Code 使用
  .gemini/: 僅 Gemini CLI 使用
  .agent/:  僅 Antigravity 使用
  .rovodev/: 僅 Rovo Dev 使用
```

### C.4 並行工作模式

```yaml
正常情況:
  sprint-status.yaml 反映整體進度
  tracking/active/ 反映當前 AI 正在執行的任務
  兩者可能不同步 — 這是預期行為

典型場景:
  情境 1: 重新審查已完成的 Story (sprint-status: done, tracking: code-review In Progress)
  情境 2: 跳過某 Story 先開發其他 (並行工作原則)
  情境 3: 跨引擎並行執行 (Claude 審查 Story A, Gemini 開發 Story B)

處理原則:
  - 優先讀取 tracking/active/ 了解當前任務
  - 參考 sprint-status.yaml 了解整體進度
  - 不要強制要求兩者同步
```

## PART 1: Token 優化配置

> **核心原理**: 每次開啟新對話會載入 CLAUDE.md、rules/、skills/ 摘要等。
> 透過精簡這些檔案，可將 session 固定開銷從 ~15,440 降至 ~2,150 tokens (-86%)。

### 1.1 Token 消耗層級分析

```
┌─────────────────────────────────────────────────┐
│ Layer 0: System Prompt (不可控)                   │
├─────────────────────────────────────────────────┤
│ Layer 1: CLAUDE.md (Global + Project + Local)    │ ← 可優化
│ Layer 2: .claude/rules/*.md                      │ ← 可優化
│ Layer 3: .claude/skills/ 摘要                     │ ← 可優化
│ Layer 4: .claudeignore 排除                       │ ← 可優化
│ Layer 5: git status 快照                          │ ← 間接影響
│ Layer 6: Workflow 執行載入                         │ ← 可優化
└─────────────────────────────────────────────────┘
```

### 1.2 Layer 1: CLAUDE.md 三層架構

```
~/.claude/CLAUDE.md          (全域，所有專案共用，≤30 行)
  ↓ 被覆寫
./CLAUDE.md                  (專案級，版本控制，≤120 行)
  ↓ 被覆寫
./CLAUDE.local.md            (本地私密，不納版控)
```

**全域 CLAUDE.md 範本** (`~/.claude/CLAUDE.md`, 目標 ≤30 行):

```markdown
# CLAUDE.md (Global Configuration)

## Language
- 所有回應、程式碼註解、文檔一律使用繁體中文 (zh-TW)
- 允許英文：程式碼、技術名詞、路徑、Git 指令、變數名

## Timestamp
- 寫入時間戳前，必須先執行系統時間查詢
- 禁止推斷、估算、使用假時間戳

## Code Quality
- 先讀再改，禁止未讀即推測
- DRY 原則，重用現有抽象
- 最小複雜度，不做額外 refactor / feature

## Reporting
- 禁止時間估算（天數、工時、完成日期）
- 以階段 (Phase) + 任務數組織結構

## File Encoding
- UTF-8, 禁止 Mojibake / 混合編碼

> Project-specific `./CLAUDE.md` will OVERRIDE these global settings.
```

### 4.7 技術債側車模式（Sidecar Pattern）

```yaml
建立（TRS-34 更新 — 配合 Debt Registry）:
  - code-review 發現 DEFERRED 項目時:
    1. 登錄 registry.yaml（Push 模式）
    2. 建立 sidecar .debt.md（三層上下文：修復指南、影響評估、商業脈絡）
    3. 在 Story 文件的 Tech Debt 表格記錄
    4. 在 Story H1 加上 🟢🚧 compound marker

消費（Pull 模式 — 自動化）:
  - create-story / dev-story 啟動時自動掃描 registry.yaml
  - 匹配 target_story 的 DEFERRED 項目 → 注入前置技術債警示
  - Sprint 規劃時檢視所有 status: pending 項目

路由規則:
  ✅ 正確: CR 延後項目留在原 Epic 內
  ✅ 正確: 登錄 registry.yaml + 建立 sidecar .debt.md
  ❌ 錯誤: 將 CR 延後項目路由至 Tech Debt Epic
  ❌ 錯誤: 僅標記 "retained" 而不登錄 registry（Silent Debt）
```

### 4.10 技術債中央登錄協議（TRS-34）

> 所有 CR 未修復的 issue 必須進入中央登錄表，禁止 Silent Debt。

**核心檔案**:
- `docs/implementation-artifacts/tech-debt/registry.yaml` — 中央登錄表（唯一真實來源）
- `docs/implementation-artifacts/tech-debt/*.debt.md` — 側車文件（DEFERRED 項目詳細上下文）

**三分類模型**（取代舊有 FIXED / DEFERRED / RETAINED）:

| 分類 | 動作 | 追蹤 |
|------|------|------|
| FIXED | CR 期間修復完畢 | 不登錄 registry |
| DEFERRED | 延後至目標 Story 修復 | 登錄 registry + 建立 sidecar |
| WON'T FIX | 明確不修復（有書面理由） | 僅登錄 registry |

**三種執行模式**:

| 模式 | 觸發時機 | 動作 |
|------|---------|------|
| Push | code-review 完成時 | CR 將 DEFERRED/WON'T FIX 項目寫入 registry |
| Pull | create-story / dev-story 開始時 | 自動掃描 registry 中匹配 target_story 的項目，注入前置技術債警示 |
| Audit | Epic 結束 / party-mode 回溯 | 掃描所有 done Story CR 報告，找出斷鏈項目 |

**多 Agent 並行注意事項**:
- `registry.yaml` 為 `exclusive_write` Hot File（見 `.agent-file-policy.yaml`）
- 寫入前必須執行 File Lock 檢查（參考 `multi-agent-parallel-execution-strategy.md` §10）
- Sidecar `.debt.md` 檔案名稱唯一，天然隔離，不需 lock

**Skill 參考**: `/example-debt-registry` SKILL.md — 完整規範（schema、規則、workflow 整合點）

## PART 8.5: Worktree 並行開發

> **適用場景**：同一引擎多開（如 5×CC-OPUS 同時推進不同 Story）。
> 每個 Agent 實例在獨立工作目錄運行，共享 `.git` 物件資料庫，檔案完全隔離。

### 8.5.1 核心概念

```
主目錄:  {project-root}/
          ├── .git/                    (共享的 git 資料庫)
          └── .claude/worktrees/
              ├── story-a/             (實例 1: branch story/story-a)
              ├── story-b/             (實例 2: branch story/story-b)
              └── story-c/             (實例 3: branch story/story-c)
```

**關鍵優勢**：每個 Worktree 有獨立的 staging area 和 branch，Agent 可自由 commit，不會互相污染。

### 8.5.2 建立 Worktree

```powershell
# 方法 1: Claude Code 內建指令（首次會觸發一次 GitHub 認證）
claude -w <story-name>

# 方法 2: 手動建立（不觸發 GitHub 認證）
git worktree add .claude/worktrees/<story-name> -b story/<story-name> HEAD
cd .claude/worktrees/<story-name>
claude
```

### 8.5.3 多 Agent 並行（5 個 PowerShell 視窗）

```powershell
# 視窗 1: claude -w ba-3
# 視窗 2: claude -w ba-6
# 視窗 3: claude -w ba-9
# 視窗 4: claude -w a10-3
# 視窗 5: claude -w d3
```

### 8.5.4 重新進入已存在的 Worktree

```powershell
cd .claude/worktrees/<story-name>
claude              # 新對話
claude --resume     # 接續上次對話
```

> **注意**：重新進入不會觸發 GitHub 認證。

### 8.5.5 完成後統一 Merge

```powershell
# 回到主目錄
cd {project-root}

# 依序 merge 各 branch
git merge story/ba-3
git merge story/ba-6     # ← 可能衝突，見 Merge 衝突 SOP
git merge story/ba-9
git merge story/a10-3
git merge story/d3

# 清理 worktree
git worktree remove .claude/worktrees/ba-3
git worktree remove .claude/worktrees/ba-6
# ... 以此類推
```

### 8.5.6 Worktree 生命週期

| 退出情境 | 行為 |
|---------|------|
| `/exit` 且無改動 | 自動刪除 worktree + branch |
| `/exit` 有改動，選 `Remove` | 刪除（改動丟失） |
| `/exit` 有改動，選 `Keep` | 保留磁碟上，可重新進入 |
| 手動清理 | `git worktree remove .claude/worktrees/<name>` |
| 查看所有 | `git worktree list` |

### 8.5.7 Merge 衝突 SOP

#### Hot File 衝突分類

| 檔案 | 衝突類型 | 解法策略 |
|------|---------|---------|
| `Program.cs`（DI 註冊） | 多個 Story 各加一行 | **兩邊都保留** — DI 註冊行無邏輯依賴 |
| `ApplicationDbContext.cs`（DbSet） | 多個 Story 各加 DbSet | **兩邊都保留** — DbSet 宣告無邏輯依賴 |
| `sprint-status.yaml` | 多個 Story 各更新狀態 | **取最新值** — 手動合併所有狀態變更 |
| `appsettings.json` | 新增 config section | **兩邊都保留** — JSON section 通常獨立 |
| `ApplicationDbContextModelSnapshot.cs` | EF Migration 自動生成 | **重新生成** — 接受任一版本後執行 `dotnet ef migrations add MergeResolve` |
| UI 共用元件（sidebar、header） | 結構衝突 | **需人工判斷** — 讀取兩邊意圖後合併 |

#### 標準解衝突步驟

```powershell
# 1. merge 時發現衝突
git merge story/ba-6
# CONFLICT (content): Merge conflict in src/.../Program.cs

# 2. 開啟衝突檔案，識別 <<<<<<< / ======= / >>>>>>>
# 3. 依上表策略決定：兩邊都保留 / 取最新 / 人工判斷
# 4. 移除衝突標記，保存檔案

# 5. 標記已解決
git add <衝突檔案>
git commit -m "merge: resolve story/ba-6 conflict in Program.cs (keep both DI registrations)"

# 6. 繼續下一個 merge
git merge story/ba-9
```

#### 「兩邊都保留」vs「需人工判斷」

```yaml
兩邊都保留（自動化安全）:
  - DI 註冊行（Program.cs: builder.Services.AddXxx）
  - DbSet 宣告（ApplicationDbContext.cs）
  - appsettings.json 新 section
  - sprint-status.yaml 不同 key 的狀態更新
  - using 語句

需人工判斷:
  - 同一函式內的邏輯修改（兩邊改同一段邏輯）
  - UI 元件結構（如同一 sidebar item 被兩邊修改）
  - 同一 CSS class 的不同修改
  - sprint-status.yaml 同一 key 的不同狀態值
  - EF Migration Snapshot（必須重新生成）
```

### 8.5.8 策略選擇指引

| 條件 | 策略 |
|------|------|
| 同引擎多開（5×CC） | **必須 Worktree** — Agent 在 worktree 內自由 commit |
| 跨引擎同目錄（CC+GC+AG+RD） | **Total Commit + File Lock** — 見 `multi-agent-parallel-execution-strategy.md` |
| 單 Agent 獨立工作 | 自由 commit，無需額外機制 |

## PART 10: Pipeline 自動化排程

> **適用場景**：需要批量執行 BMAD Workflow（create → dev → review），由主視窗 Claude 作為中控調度器。
> **詳細需求分析與根因分析**：`Claude智能中控自動化排程/pipeline-audit-token-safety.md`

### 10.1 Pipeline 架構概覽

```
┌──────────────────────────────────────────────────────────────────┐
│  主視窗 Claude（中控）                                             │
│                                                                  │
│  1. 讀取 sprint-status.yaml → 識別待處理 Stories                   │
│  2. 分批規劃（每批最多 5 個）                                       │
│  3. ★ 檢查 Token 使用率                                           │
│     ├─ < 90% → 繼續                                              │
│     └─ ≥ 90% → ⛔ 停止，輸出警示                                  │
│  4. 啟動 batch-runner.ps1 (run_in_background)                     │
│  5. 等待完成通知                                                   │
│  6. 執行 batch-audit.ps1 → 驗證 + AutoFix                         │
│  7. 解析結果 → 決定是否繼續 Batch N+1                              │
│  8. 重複 3-7 直到全部完成或 Token 不足                              │
└──────────────────────────────────────────────────────────────────┘
         │
         ▼ batch-runner（每個 Story 間隔 12 秒啟動）
┌──────────────────────────────────────────────────────────────────┐
│  每個 Story pipeline 內部：                                       │
│  ┌─────────┐  12s  ┌─────────┐  12s  ┌──────────┐               │
│  │ create  │ ────→ │  dev    │ ────→ │ review   │               │
│  │ (Opus)  │       │(Sonnet) │       │ (Opus)   │               │
│  │ 新視窗  │       │ 新視窗  │       │ 新視窗   │               │
│  └─────────┘       └─────────┘       └──────────┘               │
│  完整 BMAD          完整 BMAD          完整 BMAD                  │
│  workflow           workflow           workflow                   │
└──────────────────────────────────────────────────────────────────┘
```

### 10.2 腳本功能矩陣

| 腳本 | 功能 | 核心機制 |
|------|------|---------|
| `story-pipeline.ps1` | 單 Story 三階段管線 | 每階段 `Start-Process` 獨立新視窗 + `--append-system-prompt` 強制完整 workflow |
| `batch-runner.ps1` | 批次並行（最多 5 Story） | 間隔 12 秒啟動 + Token 安全閥 L1/L2 + 結果分類（Done/Failed/TOKEN-LIMIT） |
| `batch-audit.ps1` | 批次後驗證 + AutoFix | 7 Check：狀態同步、metadata 完整性、tracking、emoji、CR 報告驗證 |
| `epic-auto-pilot.ps1` | 整個 Epic 迴圈 | 支援 Claude + Gemini 雙引擎分工 |

### 10.3 Token 安全閥（4 層防護）

```yaml
層級架構:
  L1 — Pre-batch:
    時機: batch-runner 啟動前
    時間窗口: 前 1 小時
    動作: 整批不啟動，exit 99

  L2 — Pre-story:
    時機: 每個 Story 啟動前
    時間窗口: 批次啟動後
    動作: 不啟動該 Story，標記 TOKEN-LIMIT

  L3 — Phase Gate:
    時機: 每個 Phase 完成後
    檢查: Phase log 全文掃描 rate-limit 字串
    動作: 中止後續階段

  L4 — 事後偵測:
    時機: 批次完成後
    檢查: 結果分類
    動作: 標記受影響 Story

偵測方式:
  - 累積估算法（每階段完成後計算使用率）
  - 錯誤模式偵測（掃描 "You've hit your limit"）
  - 時間過濾（避免歷史 log 誤判）
```

### 10.4 `-p` 模式三層防護

> **背景**：`claude -p` 模式偶爾跳過 BMAD workflow 末段的 metadata 更新步驟（Story 狀態、sprint-status.yaml、tracking、emoji）。

| 防護層 | 機制 | 效果 |
|--------|------|------|
| 第 1 層 | `--append-system-prompt` 強制指令 | System Prompt 層級約束，優先保留 metadata 步驟 |
| 第 2 層 | Phase 間隔 12s + 獨立新視窗 | 每階段全新 context，不累積壓力 |
| 第 3 層 | `batch-audit.ps1` 事後驗證 + AutoFix | 漏網之魚自動修復 |

**預期效果**：失敗率從 15-20% 降至 < 1%。

### 10.5 batch-audit 7 Check 函式

| Check | 名稱 | 驗證內容 |
|-------|------|---------|
| 1 | 狀態一致性 | Story 文件狀態 vs sprint-status.yaml |
| 2 | 雙重狀態 | Story `## Story 資訊` 表格 vs sprint-status |
| 3 | Metadata 完整性 | DEV Agent / Review Agent 欄位非空 |
| 4 | File List | dev-story 完成後 File List 已更新 |
| 5 | CR 報告 | reviews/ 目錄下 CR 報告已建立 |
| 6 | Tracking 檔案 | active/*.track.md 最後修改時間驗證 |
| 7 | H1 Emoji | emoji vs 狀態一致性（🔵/🟠/🟢/🟢🚧） |

### 10.6 Pipeline 使用方式

```powershell
# 方式 1：主視窗 Claude 中控（推薦）
# 在 Claude Code 中讓 AI 讀取 sprint-status.yaml，決定批次並啟動

# 方式 2：手動單 Story
.\scripts\story-pipeline.ps1 -StoryId "qgr-e7" -Verbose

# 方式 3：手動批次
.\scripts\batch-runner.ps1 -StoryIds @('qgr-s6','qgr-m10','qgr-s7') -IntervalSec 12

# 方式 4：整個 Epic 自動化
.\scripts\epic-auto-pilot.ps1 -EpicPrefix "qgr" -MaxParallel 5

# 批次完成後驗證
.\scripts\batch-audit.ps1 -StoryIds @('qgr-s6','qgr-m10','qgr-s7')
```

### 10.7 已知限制與教訓

| 問題 | 根因 | 解法 |
|------|------|------|
| `-p` 模式跳過 metadata | Context 壓力下 LLM 自適應省略「非核心」步驟 | `--append-system-prompt` + batch-audit AutoFix |
| Write-Log 污染 pipeline | PowerShell `Write-Output` 進入函式返回值流 | 改用 `Write-Host` 輸出日誌 |
| 歷史 log 觸發安全閥誤判 | `Test-TokenHealth` 掃描全目錄 log | 改為時間過濾（pre-batch=1hr / pre-story=$StartTime） |
| Token 配額無法精確查詢 | Claude Max 無公開 API | 累積估算 + 錯誤模式偵測組合 |
